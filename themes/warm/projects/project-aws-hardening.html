AWS Security Assessment & Hardening — CloudNative Corp

Comprehensive CIS AWS Foundations Benchmark assessment and security hardening for a 400-employee cloud MSP managing 50+ enterprise client AWS environments with $15M+ annual cloud spend

Company: CloudNative Corp
Industry: Cloud Managed Services
Frameworks: CIS AWS Benchmark v3.0, NIST CSF 2.0
Trigger: External audit revealed critical misconfigurations
Date: March 2026

1. Executive Summary

CloudNative Corp is a 400-employee cloud-managed services provider operating AWS infrastructure for 50+ enterprise clients with over $15M in annual cloud spend. A recent external audit revealed critical security misconfigurations including publicly accessible S3 buckets containing client data, overly permissive IAM policies granting broad administrative access, and missing encryption on several database instances.

This project conducts a comprehensive CIS AWS Foundations Benchmark assessment, inventories 6 critical cloud assets, documents 5 specific security findings, assesses 4 cloud-specific risks, and implements 10 security controls to harden the AWS environment.

Finding 1: 138+ Credentials in Plaintext CRITICAL

  • 138+ secrets (API keys, DB passwords, encryption keys) stored outside Secrets Manager: 38 in Lambda env vars, 22 in ECS tasks, 15 in EC2 user data, 18 in S3 configs, 12 in CodeCommit history, 33 in plaintext Parameter Store
  • Remediation: Rotate all 138+ credentials within 72 hours, migrate to Secrets Manager with KMS CMK, enable auto-rotation (30-day DB, 90-day API keys), add secret detection in CI/CD (CodeGuru/truffleHog), scrub CodeCommit history

Finding 2: Shadow Infrastructure Untracked CRITICAL

  • 37 untracked resources: 14 EC2 instances (3 in public subnets, 2 on deprecated Amazon Linux 1) and 23 Lambda functions (8 without CloudWatch logs, 15 manually deployed)
  • Remediation: Tag and inventory all 37 resources, decommission Amazon Linux 1 instances, remove unnecessary public IPs, enable CloudWatch logs, enforce mandatory tagging via AWS Config rule, deploy Service Catalog for approved resource provisioning

Finding 3: CI/CD Pipeline Supply Chain Gaps CRITICAL

  • 8 CodeBuild roles with full AdministratorAccess, no security scanning (SAST/DAST/container), no artifact signing, 12/25 pipelines skip production approval, 3 build log buckets publicly accessible
  • Remediation: Restrict CodeBuild to least-privilege roles, add mandatory approval gates, secure log buckets, add scanning gates (Checkov, Semgrep, Trivy, OWASP Dependency-Check), implement AWS Signer for artifact verification

Finding 4: EKS Clusters Without Pod Security HIGH

  • 4/8 clusters on unsupported Kubernetes 1.25, no Pod Security Standards (root access allowed), 3 clusters with public API, no network policies, 12 images with Critical CVEs
  • Remediation: Upgrade to latest Kubernetes, enforce restricted Pod Security Standards, make API endpoints private, deploy Calico network policies for namespace isolation, enable ECR scanning + OPA Gatekeeper, deploy Falco for runtime monitoring

Finding 5: Access Logging Disabled on Public Resources HIGH

  • 18/35 CloudFront and 15/28 ALBs without access logging; 12 ALBs without WAF — no visibility into DDoS, brute-force, SQLi, or API abuse
  • Remediation: Enable logging on all CloudFront and ALBs, deploy AWS WAF with OWASP Top 10 rules, auto-renew SSL certs via ACM, enforce HTTPS-only, centralize logs in SIEM, deploy Shield Advanced for DDoS protection

CR-1: Supply Chain Compromise via CI/CD CRITICAL — 20

  • CI/CD deploys code to ALL 50+ client environments — compromised dependency or malicious commit could affect all clients simultaneously
  • CodeBuild has full admin access, no scanning, no artifact signing, pipelines skip approval
  • Treatment: Least-privilege build roles, 4 scanning gates, mandatory approval, AWS Signer for artifact verification, SBOM generation

CR-2: Credential Exposure — Multi-Client Breach CRITICAL — 25

  • 138+ credentials already exposed in source code history, Lambda console, and public S3 build logs
  • Database credentials provide direct client data access; shared credentials across systems multiply impact
  • Treatment: Emergency 72-hour rotation, centralize in Secrets Manager, auto-rotation, CI/CD secret detection, quarterly secrets audit

CR-3: Container Escape & Cross-Tenant Access CRITICAL — 15

  • Outdated Kubernetes with privilege escalation CVEs, pods running as root, no namespace isolation, public API endpoints
  • Container escape gives attacker access to host node running pods from multiple clients
  • Treatment: Upgrade Kubernetes, restricted Pod Security Standards, Calico network policies, private API endpoints, ECR scanning + Falco runtime monitoring

CR-4: Invisible Attack Surface CRITICAL — 16

  • 37 untracked resources, 33 public resources without logging, 12 ALBs without WAF — security team has no visibility
  • Public S3 exposure went undetected for 4 months due to monitoring gaps
  • Treatment: Mandatory tagging + Service Catalog provisioning, universal access logging, WAF on all public endpoints, GuardDuty + Security Hub org-wide

Control 1: Centralized Secrets Management with Auto-Rotation

  • All 138+ credentials migrated to AWS Secrets Manager with KMS CMK encryption
  • Auto-rotation: database credentials every 30 days, API keys every 90 days
  • CodeGuru Reviewer + truffleHog in CI/CD blocking commits containing secrets
  • Custom Config rule detects secrets in Lambda env vars, ECS tasks, and plaintext Parameter Store
  • Scrubbed all secrets from CodeCommit history via git-filter-repo; quarterly secrets audit

Control 2: Mandatory Resource Tagging + Service Catalog

  • SCP requiring 5 mandatory tags on every resource: Owner, Client, Environment, CostCenter, ManagedBy
  • AWS Service Catalog with pre-approved, security-hardened CloudFormation templates as only approved provisioning method
  • Manual console creation of EC2, Lambda, RDS, EKS blocked via SCP
  • Monthly asset inventory reconciliation: AWS Config inventory vs CMDB

Control 3: CI/CD Pipeline Security with Supply Chain Controls

  • CodeBuild roles replaced with least-privilege policies scoped to specific client accounts and resource types
  • 4 security scanning gates: Checkov (IaC), Semgrep (SAST), Trivy (container CVEs), OWASP Dependency-Check (dependencies)
  • Mandatory human approval before all production deployments via CodePipeline approval actions
  • AWS Signer for artifact signing/verification; build logs in encrypted private S3 buckets

Control 4: EKS Security Hardening

  • All 8 clusters upgraded to latest Kubernetes; Pod Security Standards enforced in "restricted" mode
  • Calico network policies isolating client namespaces; all API endpoints made private (VPC-only)
  • ECR image scanning on push; OPA Gatekeeper blocks deployment of images with Critical CVEs
  • Falco runtime monitoring for suspicious activity: unexpected processes, unsafe network connections, filesystem changes

Control 5: Comprehensive Access Logging

  • Access logging enabled on all 35 CloudFront distributions and 28 ALBs to encrypted S3 with lifecycle rules
  • VPC Flow Logs enabled on all VPCs across all 50+ accounts
  • All logs centralized in SIEM alongside application and CloudTrail logs

Control 6: AWS WAF with OWASP Top 10 Managed Rules

  • WAF deployed on ALL 28 ALBs and 35 CloudFront distributions
  • Rule groups: Common Rule Set, SQL Injection, Known Bad Inputs, IP Reputation List, Rate-Based (2000 req/5min DDoS mitigation)
  • Custom rules: geo-restriction for sanctioned countries, suspicious header blocking, per-client API key rate limiting
  • WAF logs to S3 + SIEM; monthly rule tuning for false positives and emerging threats

Control 7: DynamoDB Security Hardening

  • All 180+ tables migrated to KMS CMK encryption; PITR enabled for 35-day recovery
  • CloudTrail data events tracking ALL GetItem, PutItem, Query, and Scan operations with user identity
  • IAM policies replaced with resource-specific ARN policies (no table-level wildcards)
  • DynamoDB Streams enabled for change tracking; quarterly access review for least privilege

Control 8: Lambda Security Hardening & Runtime Management

  • All 340+ functions audited: deprecated runtimes upgraded, IAM roles restricted to exact resource ARNs
  • Execution timeouts configured (5 min for APIs, 15 min for ETL); environment variables migrated to Secrets Manager
  • Lambda function URLs audited and restricted; Config rules for runtime, timeout, memory checks
  • Monthly fleet review for abandoned functions

Control 9: Cross-Account IAM Role Trust Policy Hardening

  • All 50+ cross-account roles hardened: trust policies restricted to CloudNative identity account with ExternalId condition
  • IAM Access Analyzer across all accounts detecting externally accessible roles, S3, KMS, Lambda
  • Session duration reduced from 12 hours to 1 hour (standard) / 4 hours (platform admins)
  • All AssumeRole events logged via CloudTrail

Control 10: Automated CIS Benchmark with Security Hub & Config

  • Security Hub enabled across all 50+ accounts with CIS AWS Foundations Benchmark v3.0
  • Real-time compliance scoring; target: 95%+ (up from 38% pre-assessment)
  • AWS Config with 30+ managed rules; auto-remediation for Critical findings (public S3 removed, unencrypted flagged, SG violations restricted)
  • Weekly compliance report; monthly trend review with VP of Cloud Operations; quarterly deep-dive
