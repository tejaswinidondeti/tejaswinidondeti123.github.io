SOC 2 Type II Audit Preparation — DataFlow Analytics

End-to-end SOC 2 Type II readiness program for an AI-powered BI platform serving Fortune 500 clients

Company: DataFlow Analytics (300 employees)
Industry: AI / Business Intelligence SaaS
Frameworks: SOC 2 Trust Services Criteria, NIST CSF 2.0
Scope: 12 Controls, 4 Systems, 5 Evidence Items, 5 Tests

1. Executive Summary

DataFlow Analytics is a 300-employee company providing AI-powered business intelligence solutions to Fortune 500 clients. Enterprise customers are contractually requiring SOC 2 Type II attestation before renewing or signing contracts. This project prepares DataFlow for its first SOC 2 Type II audit by documenting 12 controls against the Trust Services Criteria, cataloging 4 in-scope systems, collecting 5 evidence items, and recording 5 control tests to simulate auditor readiness validation.

3. SOC 2 Control Catalog — 12 Controls

C-01: Logical Access Control — Role-Based Access

SOC 2: CC6.1  |  NIST CSF 2.0: PR.AA-01, PR.AA-05

  • 5 RBAC roles: Admin (8), Developer (25, read-only prod), Data Engineer (6), Support (12, read-only), Client User (12,000+)
  • Access requests via Jira Service Desk with business justification and manager approval
  • Quarterly access reviews by role owners; least privilege enforced
  • No permanent admin access — temporary privileged access via AWS SSM Session Manager

C-02: Multi-Factor Authentication Enforcement

SOC 2: CC6.1, CC6.6  |  NIST CSF 2.0: PR.AA-03

  • MFA required for all access with no exceptions
  • Admins: hardware YubiKey 5 NFC; employees: TOTP via Okta Verify
  • Adaptive MFA triggers extra verification for new devices, locations, or off-hours access
  • Login blocked if MFA not completed — no bypass possible

C-03: User Provisioning & Deprovisioning

SOC 2: CC6.2, CC6.3  |  NIST CSF 2.0: PR.AA-01, PR.AA-02

  • Automated via Okta + BambooHR integration
  • New hires auto-provisioned with correct role based on department/job title
  • Terminated employees disabled within 1 hour; all sessions revoked
  • Monthly reconciliation report comparing Okta accounts vs BambooHR employee list

C-04: Data Encryption at Rest and in Transit

SOC 2: CC6.7, C1.1  |  NIST CSF 2.0: PR.DS-01, PR.DS-02

  • At rest: AES-256 via AWS KMS customer-managed keys (RDS, S3, EBS)
  • In transit: TLS 1.3 required; TLS 1.2 and below disabled on all public endpoints
  • RDS parameter group enforces SSL (rds.force_ssl = 1)
  • KMS keys rotated automatically every year

C-05: Network Segmentation & Firewall Controls

SOC 2: CC6.6  |  NIST CSF 2.0: PR.IR-01, PR.IR-04

  • ECS runs in private subnets — traffic only through ALB + CloudFront
  • RDS in isolated private subnets — accessible only by application layer via security groups
  • AWS WAF protects against OWASP Top 10 (SQL injection, XSS)
  • Deny-by-default security groups; VPC flow logs sent to CloudWatch

C-06: Vulnerability Management & Patching

SOC 2: CC7.1, CC3.2  |  NIST CSF 2.0: DE.CM-09, ID.RA-01

  • Weekly Qualys scans on all production systems
  • Trivy container image scanning in CI/CD — Critical/High CVEs auto-block deployment
  • Patch SLAs: Critical (CVSS 9.0+) within 48 hours; High (7.0-8.9) within 14 days
  • Annual pen test by Cobalt (Nov 2025): 0 Critical, 1 High (fixed in 5 days), 3 Medium (fixed in 30 days)

C-07: Change Management Process

SOC 2: CC8.1  |  NIST CSF 2.0: PR.PS-01

  • All changes via feature branch + pull request with peer review
  • CI/CD pipeline: unit tests (85%+ coverage), Semgrep SAST, container scanning, staging QA
  • Infrastructure changes via Terraform with same review/approval process
  • Emergency changes require CISO approval + 48-hour post-change review

C-08: Security Monitoring & Alerting (SIEM)

SOC 2: CC7.2, CC7.1  |  NIST CSF 2.0: DE.AE-02, DE.CM-01

  • Datadog SIEM collecting: CloudTrail, VPC Flow Logs, Okta, application logs, RDS audit logs
  • Automated alerts: 5+ failed logins in 10 min, admin actions off-hours, large data exports, new geo logins, security group/IAM changes
  • High-severity alerts → PagerDuty → on-call engineer (30-min response SLA)

C-09: Incident Response Plan & Procedures

SOC 2: CC7.3, CC7.4  |  NIST CSF 2.0: RS.MA-01, RS.CO-02

  • 4 severity levels (P1-P4) with defined escalation paths and roles (Incident Commander, Tech Lead, Comms Lead)
  • Client notification: 24 hours for confidentiality incidents, 4 hours for availability incidents
  • Annual tabletop exercise (Jan 2026); post-incident review within 5 business days
  • Evidence preservation procedures and regulatory notification requirements documented

C-10: Business Continuity & Disaster Recovery

SOC 2: A1.2, A1.3  |  NIST CSF 2.0: PR.IR-02, RC.RP-01

  • Multi-AZ RDS failover, ECS across 3 AZs with auto-scaling
  • DR: primary us-east-1, backup us-west-2; RTO target 2 hours, RPO target 15 minutes
  • Latest DR test (Sep 2025): achieved RTO 1.5 hours, RPO 12 minutes (both under target)
  • Daily backups retained 35 days; BCP reviewed quarterly, approved by CTO

C-11: Risk Assessment Process

SOC 2: CC3.1, CC3.2  |  NIST CSF 2.0: ID.RA-01, ID.RA-05

  • Annual risk assessment for all in-scope systems using 5x5 likelihood/impact matrix
  • Risks mapped to SOC 2 criteria with documented treatment (mitigate, accept, transfer, avoid)
  • Most recent assessment: 18 risks identified, all with documented treatments
  • Tracked in GRC tool; updated for major changes, new systems, or incidents

C-12: Confidential Data Classification & Handling

SOC 2: C1.1, C1.2  |  NIST CSF 2.0: PR.DS-10, PR.DS-11

  • 3-tier classification: Confidential (client data), Internal (business data), Public (marketing)
  • Client data auto-classified as Confidential at upload; encrypted AES-256 + TLS 1.3
  • No export to local devices without DLP approval
  • Contract termination: cryptographic erasure via KMS key deletion; 90-day retention default; deletion certificate issued
