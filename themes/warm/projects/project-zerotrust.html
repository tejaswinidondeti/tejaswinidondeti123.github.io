Zero Trust Architecture Implementation — SecureBank Digital

Designed and implemented a NIST SP 800-207 Zero Trust architecture for a 750-employee digital bank processing 2.8M daily transactions and $3.2B in deposits

Company: SecureBank Digital
Industry: Digital Banking
Frameworks: NIST SP 800-207, NIST CSF, CIS Controls v8
Regulation: OCC, FDIC, FFIEC, BSA/AML
Date: March 2026

1. Executive Summary

SecureBank Digital is a 750-employee challenger bank in Charlotte, NC, offering mobile-first banking to 500K+ customers across the US. The bank processes 2.8 million transactions daily and manages $3.2 billion in customer deposits under OCC, FDIC, and Federal Reserve oversight.

Following high-profile breaches at competitor institutions and updated FFIEC guidance on authentication and access controls, the Board of Directors mandated a Zero Trust transformation. This project assessed the current perimeter-based security model, mapped 6 critical assets, identified 5 access risks, implemented 12 Zero Trust controls based on NIST SP 800-207 principles, and established 4 governance policies.

3. Access Risk Assessment — 5 Risks

Risk ID Risk Title Category Likelihood Impact Score Rating
ZT-R1 Lateral Movement After Credential Compromise Access Control 4 5 20 CRITICAL
ZT-R2 Account Takeover Through Weak Customer Authentication Application Security 4 4 16 CRITICAL
ZT-R3 Privileged Access Abuse by Insiders Access Control 3 5 15 CRITICAL
ZT-R4 Unmanaged Device Access Bypassing Endpoint Controls Endpoint Security 3 4 12 HIGH
ZT-R5 Third-Party/API Access Exceeding Minimum Permissions Vendor Risk 3 4 12 HIGH

ZT-R1: Lateral Movement After Credential Compromise CRITICAL — 20

  • Current VPN grants broad network connectivity once authenticated — compromised credential gives attacker same access as legitimate user
  • Flat network allows scanning, discovery, and lateral movement to core banking, customer DB, and payment systems
  • Phishing success rates in banking average 8-12%; VPN credentials actively traded on dark web
  • Treatment: Replace VPN with Zscaler ZPA (application-specific tunnels), implement micro-segmentation, enforce default-deny network posture

ZT-R2: Account Takeover Through Weak Customer Auth CRITICAL — 16

  • 85% of customer interactions occur via digital channels with password + SMS 2FA
  • SMS 2FA vulnerable to SIM-swapping ($68M FBI-reported losses in 2022) and SS7 interception
  • Coordinated ATO campaigns can affect hundreds of accounts simultaneously
  • Treatment: Migrate to FIDO2 WebAuthn with biometrics, risk-based step-up authentication, behavioral fraud engine

ZT-R3: Privileged Access Abuse by Insiders CRITICAL — 15

  • 45 employees have standing privileged access to production: 12 DBAs, 8 sysadmins, 15 infra engineers, 10 app admins
  • Standing access enables data exfiltration, record modification, backdoor creation, security control disabling
  • Treatment: Eliminate all standing privileges via CyberArk JIT access — request, approve, use, auto-revoke (max 4-hour windows)

ZT-R4: Unmanaged Device Access HIGH — 12

  • Current model authenticates users but does not verify device health — personal/contractor devices may lack EDR, patches, encryption
  • 60% of employees work remotely 3+ days/week; 50+ contractors use unmanaged devices
  • Treatment: Okta Device Trust requiring MDM enrollment, patched OS, active EDR, disk encryption before any access is granted

ZT-R5: Third-Party/API Over-Permissioning HIGH — 12

  • 35+ third-party API integrations authenticated with static API keys — broad permissions, no expiry, shared across environments
  • Compromised vendor API key grants attacker access with full integration permissions
  • Treatment: Replace static keys with OAuth 2.0 short-lived tokens + mutual TLS, minimum-necessary scoped permissions, quarterly access reviews

Control 1: Identity-Centric Access with Okta Adaptive MFA

  • Replaced network-based access with identity-based access through Okta Workforce Identity
  • Every request evaluated against: verified identity (FIDO2 YubiKeys / Okta Verify), device health (MDM, OS patches, EDR), risk signals (IP reputation, geo anomalies, credential breach DBs), and context (time, sensitivity, behavior)
  • High-risk access (core banking, payments, customer DB) requires step-up authentication even within active sessions
  • Serves as the Policy Decision Point (PDP) per NIST 800-207

Control 2: Zscaler Zero Trust Network Access Replacing VPN

  • Replaced traditional VPN with Zscaler Private Access (ZPA) — application-specific micro-tunnels, users never join the corporate network
  • 600/750 employees migrated (80%); 45 VPN concentrators scheduled for decommission
  • ZIA provides SSL inspection, cloud firewall, URL filtering, DLP, and sandboxing
  • Every session logged: user identity, device posture, app accessed, data volume, risk score

Control 3: Micro-Segmentation of Core Banking Platform

  • AWS Security Groups + Network ACLs + Istio service mesh for application-level segmentation
  • CBS, Customer DB, Payment Gateway each have independent security boundaries
  • Inter-service communication via mTLS through service mesh — only explicitly defined flows allowed
  • Default-deny posture: all traffic not matching defined flows is blocked

Control 4: Just-in-Time Privileged Access with CyberArk

  • Eliminated ALL standing privileged access for 45 employees with permanent admin rights
  • Request-Approve-Use-Revoke workflow: specify system, task, duration (max 4 hours), manager + security approval
  • Session recording (screen capture + keystroke logging) for all Critical system access
  • Automatic credential rotation after each session; break-glass procedure for emergencies

Control 5: FIDO2 Phishing-Resistant Customer Authentication

  • Migrating 500K+ customers from password + SMS 2FA to FIDO2 WebAuthn with biometrics
  • Mobile: Face ID / Touch ID as FIDO2 authenticators; Web: Windows Hello, macOS Touch ID, security keys
  • Risk-based step-up auth: low-risk (check balance) = seamless; high-risk (large transfer, new payee) = full re-verification
  • 35% of mobile users migrated; full migration targeted Q4 2026

Control 6: Continuous Device Trust Verification

  • Pre-access checks via Okta Device Trust: MDM enrollment, OS within 30 days of latest patch, CrowdStrike EDR active, disk encryption enabled
  • Hourly continuous checks — degraded posture during active session triggers immediate session termination
  • Contractors: Zscaler Client Connector with DLP, clipboard restrictions, watermarked screen display

Control 7: API Zero Trust with OAuth 2.0 and Mutual TLS

  • Replaced static API keys for 35+ third-party integrations with mTLS + OAuth 2.0 (1-hour token expiry)
  • Minimum-necessary scoped permissions per integration (e.g., Plaid: read-only balance + routing only)
  • API Gateway with rate limiting, request validation, anomaly detection
  • Quarterly API access review verifying permissions remain minimum necessary

Control 8: Data Loss Prevention (Classification-Driven)

  • DLP across all channels: Zscaler ZIA (outbound), M365 Purview (email), CrowdStrike (endpoint), API Gateway (responses)
  • 4-tier classification: Restricted (SSN, account numbers, SARs), Confidential (internal reports, employee data), Internal, Public
  • Microsoft Purview automated weekly scans auto-classifying untagged sensitive data

Control 9: SIEM with Zero Trust Analytics (Splunk)

  • Aggregates telemetry from Okta, Zscaler, CyberArk, CrowdStrike, AWS CloudTrail, CBS, Payment Gateway, CIAM
  • ZT correlation rules: impossible travel, privilege escalation, anomalous data access, session hijacking, device posture changes
  • 24/7 SOC with 15-minute SLA for high-severity alerts; 7-year log retention per BSA/AML

Control 10: Microsegmented Network with Default-Deny

  • All segments operate on default-deny — no traffic allowed unless explicitly permitted with business justification
  • AWS Security Groups (instance-level) + Network ACLs (subnet-level) + Istio service mesh (application-level)
  • East-west traffic encrypted with mTLS; north-south through WAF + API Gateway
  • Quarterly penetration testing validates no unauthorized paths exist between segments

Control 11: Zero Trust Maturity Metrics Dashboard

  • Tracks maturity across 5 CISA dimensions: Identity, Device, Network, Application, Data
  • Monthly reports to CISO, quarterly to Board, annual maturity assessment for OCC examiners
  • Key metrics: % apps federated (92%), phishing-resistant MFA (85%), VPN-to-ZPA migration (80%), micro-segmentation (78%)

Control 12: Regulatory Compliance Integration (FFIEC, OCC, BSA/AML)

  • ZT controls mapped to: FFIEC Authentication Guidance, FFIEC CAT domains, OCC Heightened Standards, BSA/AML monitoring
  • Regulatory-ready evidence: Okta access logs, CyberArk session recordings, Zscaler logs, maturity dashboard
  • Annual mock OCC examination conducted internally to validate readiness

5. Policy Library — 4 Governance Policies

Policy 1: Zero Trust Access Policy Approved

OwnerCISO
Review DateSeptember 30, 2026
  • No Implicit Trust: No user, device, or network location trusted by default — every request authenticated, authorized, and continuously validated
  • Identity-First: Access based on verified identity, not network location
  • Least Privilege: Minimum access for current task; standing privileged access prohibited
  • Continuous Verification: Sessions re-evaluated on changing risk signals
  • Assume Breach: Architecture assumes perimeter already compromised; controls limit blast radius
  • Device Trust: Only managed, healthy, compliant devices can access systems
  • Encryption Everywhere: All data encrypted at rest and in transit regardless of location

Policy 2: Privileged Access Management Policy Approved

OwnerCISO
Review DateSeptember 30, 2026
  • Zero Standing Privileges: No permanent admin access; all credentials vaulted in CyberArk
  • JIT Access: Specify system, task, justification, duration (max 4 hours) with manager + security approval
  • Session Recording: Screen capture + keystroke logging for Critical systems; 7-year retention
  • Credential Rotation: Automatic password rotation after each privileged session
  • Quarterly Review: All privileged access patterns reviewed to identify unnecessary access

Policy 3: Third-Party and API Security Policy Approved

OwnerVP of Payment Operations
Review DateSeptember 30, 2026
  • Mutual Authentication: All API integrations use mTLS; static API keys prohibited for new integrations
  • Minimum Scope: Each integration receives only required data fields/operations; quarterly scope review
  • Short-Lived Tokens: OAuth 2.0 tokens expire after 1 hour with refresh
  • Vendor Assessment: Annual SOC 2 review, penetration test results, security questionnaire
  • Offboarding: All credentials and access revoked within 24 hours when vendor relationship ends

Policy 4: Data Classification and Protection Policy Approved

OwnerChief Data Officer
Review DateSeptember 30, 2026
  • 4-Tier Classification: Restricted (SSNs, account numbers, SARs), Confidential (internal reports, employee PII), Internal (policies, procedures), Public (marketing)
  • Data Minimization: Store only minimum necessary; non-production uses tokenized/synthetic data
  • Retention & Disposal: 7 years for BSA/AML, 5 years for regulatory filings; cryptographic erasure after retention
  • Cross-Border: All data remains in US-based data centers; no international processing
  • DLP Enforcement: Automated controls prevent Restricted/Confidential data from leaving authorized channels
